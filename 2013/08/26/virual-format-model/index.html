可视化格式模型

发表于 2013-08-26   |   分类于 Frontend

可视化格式模型简介

本章描述可视化格式模型：即用户代理如何为视觉媒体处理文档树

在VFM中，文档树中的每一个元素根据盒模型生成零个或多个盒子，这些盒子的布局受以下因素支配：

  • 盒子的大小和类型
  • 定位方案（正常流、浮动和绝对定位）
  • 元素在文档树之间的关系
  • 外部信息（如：视口的大小、图像的固有尺寸，等）

本章定义的同时适用于连续介质和分页媒体，但 margin 可能有所区别

视口

用于连续介质的用户代理通常为用户提供一个视口（屏幕上的窗口或者其他可视区域），通过它，用户访问文档。当视口大小调整时，用户代理可以改变文档的布局。

当视口比呈现的文档小时，用户代理应提供滚动机制

包含块（Containing blocks）

在CSS2.1中，许多盒子的位置和大小计算需要考虑一个矩形框的边缘，这个矩形框称为：包含块。一般情况下，生成的盒子实际上作为包含块的子孙盒子。我们说，一个盒子为它的子孙盒子“建立”包含块。因此，短语“一个盒子的包含块”是指这个盒子所存在的包含块，而不是它生成的。

一个盒子的位置与它的包含块有关，但它不受这个包含块的束缚，它有可能溢出。

在下一章将介绍详细的包含块大小计算的细节。

控制盒子生成（Controlling box generation）

CSS 视觉格式化模型的一部分工作是从文档元素生成盒。生成的盒拥有不同类型，并在某种程度上影响可视化格式模型的行为。生成盒的类型取决于 CSS 属性 display。

块级元素和块盒

块级元素（block-level element）是在源文件中被格式化为块的元素（例如：段落）。

设置display值，使元素为块级元素：
“block”，“list-item”，“table”，“flex”，“grid”。

块级盒（block-level box）参与块级格式上下文（BFC）。每个块级元素生成一个主要的块级盒，包含子孙盒和生成的内容，它也是可以使用任何定位方案的盒。一些块级元素，例如：“list-item” 元素，除了主盒子之外，也产生额外的盒来放置项目符号，不过多数元素只生成一个主要块级盒。

除了 “table” 盒，将在后面的章节描述，和可替换元素，一个块级盒也是一个 块容器盒（block container box）。一个块容器盒只包含块级元素，或者建立一个行内格式上下文（inline formatting context），只包含行内盒。不是所有的块容器盒是块级元素，例如：非替换行内块和非替换表格单元格是块容器但是不是块级盒。

  • 在非块级格式化上下文中的块容器总是会创建新的块级格式化上下文：如display为inline-blocks, table-cells, 和table-captions所生成的盒。
  • 而自身也在块级格式化上下文中的块容器，则只有overflow不为visible的情形下才会创建新的块级格式化上下文。
  • 绝对定位和浮动的块容器则总是会创建新的块级格式化上下文。
  • display值为table或者inline-table的元素将会生成表格（table），表格内部会使用特殊的格式化方式来排布其内部元素。
  • display值为grid或者inline-grid的元素将会生成格元素（grid element），与table情形类似，它内部也是使用特殊的格式化方式来排布其内部元素，
  • display值为flex或者inline-flex的元素将会生成自适应容器（flex container），自适应容器在其内部产生自适应格式化上下文（flex formatting context）。

注意块级盒与块容器盒概念不同。 前者描述元素跟它的父元素与兄弟元素之间的表现，后者描述元素跟它的后代之间的影响。

既是块级盒又是块容器盒的，被称为：块盒。

行内级元素和行内盒（Inline-level elements and inline boxes）

行内级元素和行内级盒

行内级元素是源文档中那些不形成新的内容块的元素，内容在行内分布（例如，段落内着重的文本，行内图形等等）

以下的 display 值将使一个元素变成行内级：“inline”，“inline-block”，和“inline-table”

行内级元素生成 行内级盒（inline-level boxes），这个盒子参与一个行内格式上下文。

行内盒（inline box）

行内盒 首先是行内级盒，并且它的内容也参与它的行内格式上下文。一个非替换元素的“display”属性设为“inline”，就生成一个行内盒。

原子行内级盒（atomic inline-level boxes）

有些行内级盒（Inline-level boxes）不是行内盒，例如可替换的inline-level、inline-block、inline-table元素，它被称作原子行内级盒

它们是以一个单独的不透明的盒子来参与它们的行内格式化上下文，也就是说内容并不参与行内格式化上下文。

(img)

原子行内盒在行内格式化上下文里不能分成多行。

匿名行内盒

任何直接被包含在块级容器元素下的文本（不是在一个行内元素内），必须被作为一个 匿名的行内元素对待。

如下的HTML文本：

<p> 生成一个块盒，有几个行内盒在它里面，内容 “emphasized” 的盒子是一个行内盒，它由行内元素 <em> 生成，其他两个盒子（“Some”和“text”）是由块级元素 <p> 生成的行内盒，后者被称为匿名的行内盒，因为它们没有与之关联的行内元素。

因此匿名行内盒从它们的父级块级盒子那里继承可继承的属性，不能继承的属性使用初始值。

空白如果使用 white-space 去掉，则不会生成任何匿名行内盒。

有时如果文章上下文描述很清晰，无论匿名行内盒还是匿名块盒可以简称为 匿名盒。

有更多匿名盒子类型将会出现，当格式化“tables”时。

其他类型盒

行盒(Line boxes)

行盒由行内格式化上下文(inline formatting context)产生的盒，用于表示一行。在块盒里面，行盒从块盒一边排版到另一边。 当有浮动时, 行盒从左浮动的最右边排版到右浮动的最左边。

行盒是技术上的实现，开发者通常不用操心它。

插入盒（Run-in boxes）

插入盒，由 display:run-in 定义，由后续盒的类型决定是块盒还是行盒。可以用来在第一个段落中插入标题。

插入盒(Run-in boxes)从 CSS 2.1 标准中移除了，因为可操作的实现定义不足。 可能 CSS3 会引入，但是这是实验性质，不能用于生产环境。

“display” 属性

定位方案（Positioning schemes）

CSS 2.1 中，有三种：

  • 正常排版（块格式化、行内格式化、块框行框的相对定位）
  • 浮动
  • 绝对定位

如果一个元素是浮动、绝对定位或根元素，那么称为 排版流之外（out of flow），反之则是 排版流之内的元素（in-flow）。

元素A的排版流包括元素A和所有在排版流之内的A的子孙元素，并且这些元素的最近的排版流外祖先元素是A。

正常流（Normal flow）

格式化上下文（Formatting context）

格式化上下文指的是：初始化元素定义的环境。包含两个要点，一个是元素定义的环境，一个是初始化。

块格式化上下文（Block formatting context）

块格式化上下文是页面 CSS 视觉渲染的一部分，是块盒子及浮动所在的地方。

布局规则

  • 框会从包含块的顶部开始，一个接一个地，垂直向下地摆放
  • 两个兄弟框之间的垂直距离由 margin 属性来决定
  • 在同一个块格式化上下文中，相邻的块级框之间的垂直外边距会出现折叠(TODO: 外边距折叠)
  • 每个框的左外边距边要紧贴其包含块的左边。即使在有浮动的情景下也是如此，除非框创建了一个新的块格式化上下文(在这种情况下该框可能会为了避开浮动框而变窄)

何时创建

下列情况将创建一个块格式化上下文：

  • 根元素或其它包含它的元素
  • 浮动 (元素 float 不为 none)
  • 绝对定位元素 (元素 position 为 absolute 或 fixed)
  • 内联块 inline-blocks (元素 display: inline-block)
  • 表格单元格 (元素 display: table-cell，表格单元格默认属性)
  • 表格标题 (元素 display: table-caption, 表格标题默认属性)
  • 元素 overflow 不为 visible
  • 弹性盒 flex boxes (元素 display: flex 或 inline-flex)

一个块格式化上下文包含这个元素的所有的后代元素，非后代元素将创建新的块格式化上下文。

块格式化上下文对定位 (参见 float) 与清除浮动 (参见 clear) 很重要。定位与清除浮动只适用于同一块格式化上下文。浮动不会影响其它块格式化上下文的布局，清除浮动只是清除同一块格式化上下文里在它前面的浮动。在块格式化上下文中相邻的块级元素的垂直外边距会折叠( collapse )。

在 CSS3 中，对块格式化上下文这个概念做了改动，将 “Block formatting context” 叫做 “flow root”。

对于触发方式也做了修改，更加准确：
The value of ‘position’ is neither “static” nor “relative”.

注意，”display:table” 本身并不产生 “block formatting contexts”。但是，它可以产生匿名框 ， 其中包含 “display:table-cell” 的框会产生块格式化上下文。 总之，对于 “display:table” 的元素，产生块格式化上下文的是匿名框而不是 “display:table”。

行内格式上下文

相对于块格式化上下文，在行内格式化上下文中，盒子( boxes )一个接一个地水平排列，起点是包含块的顶部。包含那些盒子的长方形区域，会形成一行，叫做行盒。

布局规则

  • 水平方向上的 margin，border 和 padding 在盒子之间得到保留，竖直方向 margin 无效。
  • 盒子在垂直方向上可以以不同的方式对齐：它们的顶部或底部对齐，或根据其中文字的基线对齐。
  • 行框的宽度由它的包含块和其中的浮动元素决定。高度的确定由行高度计算规则决定

行内框的分割

如果一个行内框超出包含它的行框的宽度，它会被分割成几个框，并且这些框会被分布到几个行框内。如果一个行框不能被分割（例如， 行内框只包含单个字符，或者语言特殊的断字规则不允许在行内框里换行，或者行内框受到带有 “nowrap” 或 “pre” 值的 ‘white-space’ 特性的影响），这时，行内框会溢出行框。

如果一个行内框被分割，margin、padding 和 border 在所有分割处没有视觉效果。

行内框还可能由于双向文本处理（bidirectional text processing）而在同一个行框内被分割为好几个框。

行内框的忽略

  • 不包含文本
  • 保留空白符
  • margin/padding/border 非0的行内元素
  • 其他常规流中的内容(比如，图片，inline blocks 和 inline tables) 并且不是以换行结束的行框

以上必须被当作零高度行框对待，如果为了其它的一些目的，则必须以其不存在来对待。
