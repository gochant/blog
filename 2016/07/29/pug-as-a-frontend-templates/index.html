Pug 作为纯前端模板提高复用性

发表于 2016-07-29   |   分类于 Frontend

本文主要介绍一种使用 nodejs 下模板引擎 pug 作为前端开发时工具的方法，同时以这种方法衍生的一个库：pukit

如果你的项目是如下的情况，那会格外有用

  • 完全前后端分离，运行时没有后端参与渲染模板
  • 没有采用任何一种能够自定义组件标签的框架（Polymer、React、Angular、Vue…）
  • 大量使用无侵入式JavaScript（常借助HTML标签属性）方式初始化界面控件

语言选择

纯HTML写页面有以下痛点

  • 代码无法复用，大量Ctrl+C/V；
  • 编码量较大，对开发人员不友好；
  • 只要基础框架HTML结构调整，所有用到的地方都要调整；
  • 不够灵活，无法根据数据配置动态生成界面

解决以上痛点，界面语言应具备以下特性：

  • 自定义函数（元素）
  • 界面扩展/继承
  • 外部参数传入

轻量级的前端模板基本上是标配，但是例如 Underscore、JQuery tmpl 等前端模板仅满足最后一点或勉强满足前面几点，但实现方式一点都不优雅，作为局部界面的生成还是可以的，如果开发人员用该类模板来编写整个界面，还是捉襟见肘

在综合比较了一些JavaScript的前后端模板后（主要包括：handlebars、haml、Swig、Jade……），最终选择了 Jade，当然它现在改了个新名字：pug

解决方案

大多数模板引擎的使用都包含 compile 和 render 两个步骤，因此可以细分为以下使用场景，注意我们这里所说的动态都是纯前端的，即后端不参与任何渲染的情况：

1. 动态 compile 及动态 render

这种模式具有最大的灵活性，前端获取 pug 模板，即时编译然后渲染，开发人员的任何改动都可以马上呈现到界面上，这种模式官方没有给例子，也没验证 pug 的服务器端运行环境能否移植到浏览器端，因此不予考虑。而且在现实开发角度，实现纯前端的动态编译也没多大的实用价值

2. compile 后动态 render

这种模式就是通过后端工具将 pug 模板预编译成前端可执行的 js 代码，前端获取这段代码（就是一个 js 函数）传入参数执行。这种模式也比较灵活，可以把需要动态的界面提成外部参数，运行时动态加载渲染

pug 提供了一个方法：compileClient 实现这种模式，它将 pug 模板编译成前端 js 代码，这里面有些问题（keng）需要说明一下：

外部 runtime

编译后的客户端代码需要依赖一小段 runtime 代码，默认编译时会包含这段代码，可以通过编译时传入 inlineRuntimeFunctions 为 false 指定不包含这段代码（文档里面没写这个 option），这样你可以外部引入一个公共的 runtime 代码。在我写这个文章时，官方 readme 文档指定的下载链接所下载的 runtime 文件在新的 pug 环境下要报错，缺少了 pug.classes 这个方法，因此需要自己从 pug-runtime 项目中下载，并进行适当包装以适应浏览器环境

mixin 管理

如果在模板中使用了大量 mixin，会造成编译后的文件非常大，尤其表现在当你把所有 mixin 放进一个公共文件引入的情况。原因在于 pug 为了保证生成的代码能够独立运行，包含了所有 mixin 的代码，唯一的做法就是在你的文件里面只 include 该用的 mixin，但实践证明，这种做法维护起来相当麻烦，很容易就遗漏或过多引入用到的 mixin

看了下 pug 的代码， pug 对 mixin 的代码生成还是做了一些优化，它在语法分析（parser）中会识别未使用的 mixin 并标记，在代码生成时不生成它，只不过它识别未使用的 mixin 用的方法是在词法分析（lex）生成 token 过程中查找 mixin call（以 +[mixinName] 开头）的方式，而这个 call 有可能包含在其他未使用的 mixin 中的，因此它无法准确找到当前文档未使用的 mixin，最终还是会有大量冗余

我尝试将 mixin 生成独立文件，而其他模板编译不生成 mixin 的代码，根据它默认的代码生成器，面临两个问题：一是 mixin 导入，这个可通过替换 mixin 变量的声明代码解决，另一个是传递生成结果的输出，这个是个棘手的问题，因为所有的代码块和 mixin 都是通过闭包的方式读取或写入一个叫 pug_html 的局部变量完成结果输出的，因此独立文件的 mixin 用的是一个 pug_html，而使用 mixin 的文件用的是另一个局部 pug_html，要么进行代码替换使用全局变量，要么修改它的源代码，都不是什么好的方案，如果后面有富余的时间再做研究

3. compile 后直接 render

通过后端工具编译并直接在后端调用，最终生成 html 或其他轻量级的前端模板，前端最终获取普通文本

这种方式几乎是最优方案，效率和传输文件大小都跟手写 html 相差无几。唯一的缺陷是不支持运行时动态传参生成界面

如何实施

一般情况下采用方案3，界面的动态化可使用其他轻量级的前端模板或MVVM的方案，在一些重度需要动态化的场合可考虑使用方案2

鉴于上面提到的方案 2 的问题，因此建议需要管理大量模板并且还没有很好分离 mixin 的方案之前，尽量少用该方案，实际测试所生成的代码至少是方案3 生成的纯 html 的几倍

自动化工具

实现方案3，工具只需要满足模板编译和监听即可，而方案2还需要实现在渲染编译结果时动态传入一些参数

后记：Pug 中有用的语言特性

pug 中有几个特性是我觉得比较重要的，如果不太熟悉可以看一下

Mixin

这个特性可以用来实现自定义元素

Includes

这个特性用来提取公共代码，通常用来放置 mixin，暂时不支持 include 传入参数，因此无法实现类似于 partial view 的功能

Extends

这个特性可以用来实现公共布局

Attributes

这个参数作为 mixin 的隐含参数，用于方法签名之外的标签属性传递，比较优雅

Block

实现自定义标签嵌套的关键性语法
