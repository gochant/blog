记一次 ODP.NET 问题排查

发表于 2016-08-11   |   分类于 .NET

最近项目组遇到很多 Oracle 开发的问题，由于近来后端人员极度缺乏，同事也叫我看了一下关于 Oracle 连接相关的问题，通过查资料和问题排查，我把几个同事碰到的几类问题总结一下记在这里：

如果条件允许，使用托管 dll

Oracle 对于 .NET 开发者提供了官方的 Oracle Data Access Components (ODAC)，主要是用的 Oracle Data Provider for .NET（odp.net），这个又分两种：托管的和非托管的。出问题的那个项目用的是非托管的 dll，如果使用托管的版本，就不会出现这些杂七杂八的问题，以下问题记录都是针对非托管的版本，也就是 Oracle.DataAccess.dll 这个库

Oracle.DataAccess.dll 找不到依赖项

这个是技术支持部署遇到的问题，看了下是在 IIS 上部署网站或服务的时候，应用程序域开启了启用 32 位程序，禁用该选项即可解决问题。这个是比较常见的问题，主要是搞不清什么时候应该启用 32 位，所以凡是部署网站都启用 32 位，似乎也没啥问题，但是遇到非托管 dll 就悲剧了，因为程序 odp.net 使用的是x64的非托管dll

关于 32 位 和 64 位

开发时编译生成的版本基本都是 any cpu 的，这个结果就是生成的程序集在 32 位进程中被 JIT 成 32 位代码，64 位进程中被 JIT 成 64 位代码，IIS 启用 32 位就是将该应用程序域进程（w3wp）搞成了 32 位环境，虽然也能运行，但进行大内存占用的行为时就会比较有局限性，另外如果程序内使用了 非托管的 dll，这个就必须注意版本了，32位的在32位上跑，64位在64位上跑

Oracle.DataAccess.Client.OracleException: The provider is not compatible with the version of Oracle client

provider 与 Oracle client 版本不兼容，这个是一开始同事叫我看的问题，说的是安装了 Oracle 11g 后就会出现该问题，后来看了下我们程序用的 odp.net 的版本，是 12c 的，会不会是因为 Oracle 11g 把配置给污染了呢？后来我自己找了个干净的环境安装了 11g，并布上我们的网站和服务，能够正常运行，说明不单纯是 11g 的问题，后来翻了一遍官方文档，发现了一些线索，这篇文档提到 Oracle.DataAccess.dll 与 非托管 dll 版本必须要匹配的问题，并介绍了查找路径的规则：

The Oracle.DataAccess.dll searches for dependent unmanaged DLLs (such as Oracle Client) based on the following order:

  1. Directory of the application or executable.
  2. DllPath setting specified by application config or web.config.`
  3. DllPath setting specified by machine.config.

  4. DllPath setting specified by the Windows Registry.

    HKEY_LOCAL_MACHINE\Software\Oracle\ODP.NET\version\DllPath

  5. Directories specified by the Windows PATH environment variable.

原来是的 dllpath 的问题，最粗暴的方法是在 web.config 中配置 dllpath，另外也可去查 machine.config 和 注册表中是否配置正确

比较奇怪的是按理来说优先去应用执行目录下查找 dll，应该不会发生这个问题，所以还存在其他因素我没找到，但后来无意中搞好后，原有的问题环境反而无法复现了

依赖注入失败，程序调用 NullReference

这个主要是缺少 dll 或放入了不正确的版本 dll 造成的，因为 odp.net 主要是两个版本：2 和 4，这个倒是个比较清晰的问题
